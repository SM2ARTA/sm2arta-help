System Overview


SM2ARTA™ (System for Materials Management and Asset Tracking) is a cloud-based platform developed to support material planning, asset tracking, and logistics coordination for large-scale events.


The system allows users to:


  • Structure and store event information in a single location;
  • Consolidate material requirements across multiple venues and functional areas;
  • Plan, monitor, and optimize procurement, storage, and dissolution workflows;
  • Track materials and assets throughout their event lifecycle;
  • Generate real-time dashboards and operational reports for decision-making;
  • Easily manage continuous changes in material requirements, ensuring that all updates are instantly reflected across relevant planning, reporting, and logistics modules.


SM2ARTA incorporates strict access controls and comprehensive data protection measures to ensure that all information remains secure, accurate, and reliable.


The platform aligns with the key principles of SOC 2 Type II and ISO 27001:2022, demonstrating its commitment to industry-standard security and operational best practices, although formal certification has not yet been obtained.


All core service providers — OVH, Cloudflare, and SendGrid — maintain active certifications, ensuring that the underlying infrastructure meets recognized international security and compliance standards.


System Architecture and Technology Stack


SM2ARTA uses modern, open-source technologies designed for high performance, reliability, and security:


  • Back-end Framework: PHP 8.1 (Laravel 10);
  • Database Engine: PostgreSQL (12.1-alpine);
  • Operating System: Linux (Ubuntu);
  • Containerization: Dockerized services (application, database, web server, and supporting components).;
  • Web Server: NGINX within Docker container;
  • Email Relay: SendGrid — used exclusively for sending password reset links;
  • Authentication: Laravel Sanctum (API sessions), Spatie Permission (RBAC), Lab404 Impersonate (login as user).


Each service operates within its own Docker container, providing strong isolation, scalability, and ease of maintenance across development, staging, and production environments.


Separate Docker networks isolate the application, database, and Vault services, with no direct internet access to internal containers.


Hosting and Infrastructure


SM2ARTA is hosted on OVH Cloud Virtual Private Servers (VPS) located in the OVH Frankfurt (Germany) data center.


The hosting environment offers:


  • ISO/IEC 27001–certified and GDPR-compliant infrastructure;
  • Encrypted SSD storage with automated backups;
  • OVH Edge Network Firewall for inbound traffic filtering;
  • Continuous uptime and performance monitoring;
  • Secure access through encrypted SSH and VPN connections.


The German hosting location ensures data residency within the European Union, compliant with EU data protection regulations.


Access Control


SM2ARTA uses a Role-Based Access Control (RBAC) model aligned with the principle of least privilege, ensuring that each user can access only the data and functions required for their role.


User Roles:


  • Material Planning Manager – System administrator with full access to all modules and configuration settings;
  • Functional Material Planner – Functional area representative with permissions to manage materials within their assigned projects, venues and spaces;
  • Venue Logistics Manager – Advanced user with controlled access to Material Planning and Asset Tracking modules for specific venues.


Access Management Controls:


  • Segregation of Duties (SoD): Access approvals and account implementation performed by different roles to prevent conflicts of interest;
  • Access Reviews: User permissions reviewed regularly to ensure ongoing accuracy and relevance;
  • User Deactivation: Offboarded or inactive accounts promptly deactivated to prevent unauthorized access;
  • Password Policy: Minimum 8 characters, including uppercase, lowercase, numbers, and symbols.
  • Secure Credential Handling: Plain text passwords are never stored or transmitted; passwords are hashed at rest, and are encrypted in transit;
  • Two-Factor Authentication (2FA): Implemented via a one-time email verification code required after password login, providing an additional layer of account security;
  • Pre-Activation Workflow: User accounts are configured and reviewed before activation;
  • Audit Logging: All login attempts are automatically recorded for traceability.


Application Security


SM2ARTA maintains a secure, multi-layered architecture designed to protect the confidentiality, integrity, and availability of data across all environments.


The platform does not expose unauthenticated public APIs; application APIs are protected by Laravel Sanctum and RBAC. Outbound-only integration is established with SendGrid, used exclusively for password reset and verification emails. All communications are encrypted using TLS 1.2 or higher, ensuring secure data exchange.


Data Encryption


  • In Transit: All data transmitted between clients and servers is protected by TLS 1.2 or higher with AES-256 encryption, HSTS enforcement, and secure cookies;
  • At Rest (Field-Level Encryption): Sensitive fields such as names, emails, and phone numbers are encrypted using CipherSweet;
  • Encryption Key Management: Keys are automatically managed by HashiCorp Vault OSS, with annual key rotation executed via cron job. No human access is permitted. Vault operates entirely within the VPS and is not connected to external KMS services.


Web Application Firewall (WAF)


All inbound traffic to SM2ARTA is routed through Cloudflare Pro, which acts as a secure global proxy between users and the application.


Cloudflare provides:


  • Protection against web vulnerabilities (SQLi, XSS, CSRF, etc.);
  • DDoS and bot mitigation;
  • OWASP Top 10 rule enforcement;
  • IP masking of backend servers (origin IPs remain hidden);
  • Enforced HTTPS and TLS encryption;
  • Rate limiting and geo-blocking for additional security.


Only verified and filtered traffic from Cloudflare’s global edge network can reach the SM2ARTA servers, ensuringMaybe both security and high availability.


Anti-Spam & Email Security


SM2ARTA does not receive inbound email.


All outbound email traffic is routed through SendGrid, where messages are authenticated with SPF/DKIM and protected by a DMARC policy..


To minimize risk, no attachments or free-text emails are permitted — only system-generated notifications and verification messages are sent.


File Security


All uploaded files are automatically scanned using ClamAV. Suspicious files are quarantined, and antivirus signatures are updated regularly to ensure continuous protection.


Data Integrity and System Protection


SM2ARTA enforces strict data integrity safeguards to maintain the consistency and accuracy of operational information.


Key data entities — such as Events, Users, Projects, Venues, and Materials — cannot be deleted once linked to other entries.


This design prevents the accidental removal of dependent data (e.g., planning entries, material allocations, or asset transactions).


Change Management and Deployment Control


SM2ARTA follows formal change management procedures to ensure that only approved and verified updates are deployed to production environments. All configuration and code changes are version-controlled and tracked through Git, ensuring full transparency and traceability.


Change Management Process:


  • All code updates undergo peer review and approval before deployment;
  • Automated CI/CD pipelines manage testing, validation, and controlled releases;
  • Deployment logs and version tags provide complete release history and traceability;
  • Rollback procedures are in place to quickly restore prior versions if required.


Environment and Data Isolation


Each SM2ARTA instance operates within an isolated environment to prevent unauthorized access and data overlap:


  • Dedicated PostgreSQL databases and unique encryption keys per instance;
  • Separate Docker containers for production, staging, and testing environments;
  • Only anonymized or synthetic data is used in non-production environments.


This segregation ensures that testing and development activities cannot impact live systems or expose sensitive data.


Data Segregation


Each tenant environment is fully isolated through architecture-level controls:


  • Separate Docker networks and PostgreSQL databases for every tenant;
  • No shared tables, keys, or data interconnections between environments.


This structure enforces logical separation and eliminates the risk of cross-tenant data exposure.


Logging and Incident Response


SM2ARTA maintains comprehensive, multi-layered logging and monitoring to ensure visibility, traceability, and rapid response to security events.


  • Source Control Layer (Git):

Git logs record every code change, including the author, timestamp, and commit message.


  • Network Layer (Cloudflare):

Traffic logs capture inbound requests, firewall actions, rate-limiting events, and blocked threats such as SQL injection or bot activity.


  • Infrastructure Layer (OVH VPS):

System-level logs record authentication attempts, SSH sessions, resource utilization, and system updates.


  • Application Layer (SM2ARTA):

All user interactions, logins, data exports, and administrative operations are logged within the application for full traceability.


Additional controls include:


  • Logs are securely stored and retained in line with compliance and operational requirements;
  • Access to application layer logs is restricted to authorized Material Planning Managers;
  • Automated alerts are generated for failed login attempts and anomalous access patterns;
  • Each detected or reported incident is documented, tracked, and analyzed to identify root causes and prevent recurrence.


System Maintenance and Monitoring


SM2ARTA  is continuously monitored for system health, performance, and security.


All virtual servers and containers are configured in line with industry best practices to minimize vulnerabilities:

  • Minimal Ubuntu 22.04 LTS images;
  • SSH key-based authentication only (no password logins);
  • Containers run under non-root privileges;
  • Monthly security updates and patch management applied to all components.


System and application updates are checked and applied monthly, with critical patches prioritized and implemented within 72 hours of release.


In addition, automated vulnerability scans are periodically performed to identify potential security risks in the application.


These authorized tests are conducted using approved third-party tools such as HostedScan.io and OWASP ZAP, under controlled conditions and with restricted test accounts.


All vulnerability testing is carried out in a dedicated test environment, separate from production servers, to ensure that live operations are never affected.
